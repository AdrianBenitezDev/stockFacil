Implementacion de suscripciones con Mercado Pago en registro de negocio

Objetivo
- Implementar suscripciones recurrentes con Mercado Pago (no pago unico).
- Usar los planes actuales que ya existen en Firebase (coleccion "planes").
- Mantener alta automatica para plan gratuito ("prueba").
- Enviar emails desde backend con estado de suscripcion, usando admin@stockfacil.com.ar.

Decisiones de arquitectura
- Proveedor: Mercado Pago.
- Modelo de cobro: suscripcion recurrente (preapproval / auto_recurring).
- Fuente de planes: Firestore "planes" (backend toma monto y configuracion desde ahi).
- Email saliente: backend (Cloud Functions) usando la integracion actual con Resend y remitente admin@stockfacil.com.ar.

Criterio de activacion de cuenta (obligatorio)
- Plan "prueba": alta inmediata.
- Plan pago: alta del tenant/usuario solo cuando exista evidencia de cobro inicial exitoso (primer payment approved asociado a la suscripcion).
- No habilitar acceso solo por "authorized" si no hay confirmacion de cobro.

Flujo funcional propuesto
1) Usuario completa formulario en registro y elige plan.
2) Si plan = "prueba":
   - Flujo actual: registerEmployerProfile crea usuario + tenant.
   - Se envia email de verificacion (flujo existente).
3) Si plan != "prueba":
   - Frontend llama createSubscriptionCheckout.
   - Backend valida payload + token Firebase.
   - Backend relee plan en Firestore (id, precio, moneda, intervalo).
   - Backend crea pending_registrations/{registrationId}.
   - Backend crea suscripcion en Mercado Pago (preapproval) con campos completos.
   - Frontend redirige al init_point de Mercado Pago.
4) Usuario autoriza la suscripcion en Mercado Pago.
5) Mercado Pago envia webhook (eventos de preapproval y/o pagos).
6) Backend valida firma, consulta API de Mercado Pago y aplica idempotencia.
7) Si hay primer pago aprobado:
   - Backend finaliza alta de usuario/tenant.
   - Guarda datos de suscripcion en tenant y usuario.
   - Envia email de "suscripcion activa" al cliente y copia a admin@stockfacil.com.ar.
8) Frontend consulta getSubscriptionStatus al volver del checkout y muestra estado real.
9) Job programado de reconciliacion revisa pendientes y corrige desfasajes por webhook perdido/tardio.

Requisitos de planes en Firebase
- La web debe usar los planes actuales, pero el backend necesita campos parseables para suscripcion.
- Campos minimos recomendados por cada plan en "planes/{planId}":
  - activo: boolean
  - titulo: string
  - precioMensual: number (no string con simbolo)
  - moneda: "ARS" (u otra)
  - intervalo: "months"
  - intervaloCantidad: 1
- Si hoy el precio esta como texto (ej: "$10.000"), agregar campo numerico adicional sin romper UI actual.

5) functions/src/mercadoPagoWebhook.js (nuevo)
- Recibe eventos de Mercado Pago.
- Verifica firma (x-signature) usando req.rawBody (bytes crudos del request).
- Consulta recurso real en API MP (no confiar ciegamente en payload entrante).
- Resuelve external_reference -> pending_registration.
- Idempotencia estricta (no reprocesar eventos ya aplicados).
- Actualiza estado de suscripcion:
  - pending_authorization
  - active
  - paused
  - cancelled
  - payment_rejected
- Si llega primer payment approved:
  - Ejecuta finalizeEmployerRegistration una sola vez.
- Dispara envio de email segun transicion de estado.

- Revalidar que email del payload coincida con email del token Firebase.

Pruebas minimas
- Registro con plan prueba (sin cambios).
- Registro con plan pago + primer pago aprobado -> alta creada.
- Suscripcion autorizada pero sin cobro -> no habilitar alta.
- Suscripcion rechazada/cancelada.
- Renovacion con pago rechazado y email de aviso.
- Reenvio duplicado de webhook.
- Webhook perdido y recuperacion por reconciliacion.
- Usuario que intenta crear multiples checkouts simultaneos.
- Corte de red entre redirect y retorno.
